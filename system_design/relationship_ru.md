# Связи
## Описание
Между [объектами](../objects_ru.md) внутри [системы](../Fundamentals/System) всегда существуют **связи**, и эти связи являются одним из двух основных понятий всего системного дизайна, наравне с объектами.

Связь само по себе является [бинарным отношением](../Fundamentals/relationship_ru.md) , поэтому у каждой связи могут быть свои характерные свойства как асимметричность (аггрегация) или транзитивность (наследование).

Самыми распространёнными (*базовые*) связями при дизайне систем являются:
- **Ассоциация** — один объект *имеет знание* (ссылку) про другой объект
- **Агрегация** — один объект *содержит в себе* другой объект
- **Композиция** — один объект содержит в себе другой объект, который *не может существовать* сам по себе
- **Наследование** — один объект, называемый ребёнком, *получает свойства* от другого объекта, называемого родительским

Остальные связи, специфичные для конкретных систем, часто можно представить в виде одного из типов связей, описанных выше.

## IT пример
В качестве связей в нашем чайном Starbucks могут быть:
- Пользователь может иметь один или несколько активных заказов
- Заказ знает про совершившего его пользователя
- Ресторан нашей сети может иметь одного или несколько барист, обслуживающих клиентов
- Бариста присвоен к одной точке сети в конкретный день недели
- Пользователь имеет свой внутренний кошелёк, и кошелёк не может существовать без пользователя
- Заказ должен иметь один или несколько чаёв внутри себя
- Пользователь может или иметь, или не иметь статус в программе лояльности, в зависимости от авторизации
- Бариста и клиент могут авторизоваться в одно и то же приложение, но для них будут разные возможности (наследование от 'авторизованный')

Так как текстовое описание является не самым наглядным, то в системном дизайне придумали разные стандарты для выражения данных связей через базовые в виде схем и диаграмм.
